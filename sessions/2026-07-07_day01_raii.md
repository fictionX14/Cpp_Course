# Day 01 — RAII & Smart Pointers  (2026-07-03)
Phase 0 · ~30 min

## Learned
- **Stack vs heap.** Stack = automatic storage, freed when scope exits (plates swept off). Heap = manual, rented via `new`, stays until you `delete` it. Heap buys you runtime-sized / longer-lived memory, but every `new` needs a matching `delete` on *every* exit path.
- **Why manual memory is fragile.** An early `return` skips the `delete[]` → leak. Worse: an **exception** thrown between `new` and `delete` unwinds the stack right past the `delete` → leak. Exception exit paths are *invisible*, so you can't hand-write enough deletes to be safe.
- **RAII (Resource Acquisition Is Initialization).** Wrap the resource in a **stack** object: acquire in the constructor, release in the destructor. A stack object's destructor is *guaranteed* to run on every exit path (including exception unwinding) → cleanup is automatic and un-forgettable. Built a hand-rolled `IntArray` wrapper to prove it.
- **The STL already does this.** You rarely hand-write RAII classes: `std::vector`/`std::string` (buffers), `std::unique_ptr`/`std::shared_ptr` (heap objects), `std::lock_guard` (mutexes), `std::fstream` (files) are all RAII. Templates make them generic → one class covers all cases, no explosion. Modern rule: prefer containers + smart pointers; almost never write raw `new`/`delete`. `std::vector<int> data(100);` was the real fix to our leak example.

## Q&A / Quiz  → score: solid
- Q1 (find the leak): ✅ correct — early return skips `delete[]`. Vocab fix: that's an *early return*, not an "exception clause."

- Comprehension (why RAII survives an exception): ✅ correct, unprompted — stack object's destructor runs during unwinding; its `delete[]` frees the memory.
